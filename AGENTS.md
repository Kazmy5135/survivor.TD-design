# Roguelike Build 设计工作区：Agent 输出规则

本目录用于设计 Roguelike 局内 Build 套路、组件关系、投放结构与数值拆解。所有 Agent 在本目录开始任务时，必须先读取 [MVP 战斗设计总控](./设计总控.md)，再读取 [事实依据索引](./03_设计边界与规范/00_事实依据/规范_事实依据索引_V1.0.md)，并遵守 [Agent输出规范](./00_Agent输出规范/Agent输出规范.md)。

## 设计总控同步

- `设计总控.md` 是查询当前设计节点、阻塞原因、文档状态和待审核队列的唯一进度入口。
- 每次进入本工作区处理任务，必须先读取总控；发现总控与具体文档不一致时，先修复同步，不继续下游设计。
- 任何受控文档的新建、正文修改、状态变化、改名、移动、删除、版本替换或依赖变化，都必须在同一次修改中同步总控。
- 同步内容至少包括文档登记、更新时间、变更摘要、当前节点、阻塞原因、审核队列和状态统计中受到影响的部分。
- 修改其他文档而未同步总控时，本轮修改视为未完成，不得宣布交付或推进下一节点。
- 总控因同步本身产生的修改不触发递归同步。

## 文档状态与设计门禁

- 设计文档状态只允许 `Draft`、`Review`、`Accepted`。
- 只有 `Accepted` 文档可以作为下游事实依据、前置条件、关联规范或开发输入。
- `Draft` 和 `Review` 只能用于讨论、审核、方案比较与历史追溯，不能解锁下一设计节点。
- Codex 可以把 Draft 整理为 Review，但只有设计负责人可以明确批准为 Accepted。
- 当前过渡期内，现有受控设计文档全部为 Review；在 [文档状态与设计门禁](./03_设计边界与规范/规范_文档状态与设计门禁_V0.1.md) 和 [MVP 战斗设计 D1-D3 规划路线](./02_过程与决策存档/2026-07-21_MVP战斗设计D1-D3_需求拆解.md) Accepted 前，不开始 D1。
- 后续讨论如果需要 D1 尚未定义的语义或规则，必须告诉设计负责人缺失项并暂停当前节点，先回到 D1 完成 Draft→Review→Accepted，再恢复后续工作。

## 审核材料的对话交付

- 凡是需要设计负责人审核、评论或批准的文档，Agent 必须把待审核正文直接发在对话中，不得只发送文件链接、文件路径或要求设计负责人自行打开文件。
- 对话中的审核稿必须保留原文的标题、章节、表格、状态标记和待确认项，使设计负责人可以直接引用具体段落发表评论。
- 文档过长时，可以按章节拆成连续多条对话消息，但不得用摘要替代待审核正文，也不得因拆分而省略内容。
- 本地文件链接可以在正文之后作为辅助定位信息提供，但不能替代对话中的完整审核内容。
- 设计负责人给出评论后，Agent 应将修改落实到本地文档、同步 `设计总控.md`，再按上述方式重新提交需要复审的正文。

## 事实依据工作流

对话不是长期事实库。任何会影响后续设计的项目事实、用户确认结论或规则变更，必须先保存到本地事实依据或设计规范中，再用于正式 Build。

### 开始任务前

1. 先读 [事实依据索引](./03_设计边界与规范/00_事实依据/规范_事实依据索引_V1.0.md)。
2. 根据索引读取相关文档的状态；只有 Accepted 内容可以进入确定性设计输入。
3. 根据索引中的“回答范围”读取与本任务相关的本地事实原文，不要求无差别加载全部材料。
4. 再读取 `03_设计边界与规范/` 中相关派生规范；Draft/Review 只能作为待审内容。
5. 最后查阅相关正式 Build 和过程存档，避免重复提出已解决问题，但不得把非 Accepted 结论当作事实。

### 事实等级

- **F0：用户明确确认的当前决策**。保存在 `规范_事实_用户确认Build设计决策_*.md`，在对应 Build 设计范围内优先级最高。
- **F1：已批准的上游设计文档**。例如游戏愿景和核心战斗循环，是产品体验与战斗规则的权威来源。
- **F2：当前系统边界文档**。用于主动技能、被动技能、神器、进化和随机结构。
- **D：派生设计规范**。由 F0～F2 推导，必须链接依据；不得覆盖上游事实。
- **R：参考候选**。可以提供思路，不能作为项目事实引用。V1.2 神器共享网络版属于此级。

当不同来源冲突时：

1. 先检查状态；非 Accepted 不参与权威冲突裁决；
2. 再按事实等级判断；
3. 同等级以更新时间更晚且范围更具体者为准；
4. 无法判断时记录冲突并向用户确认，不得静默选择；
5. 用户给出新结论后，必须在同一任务内更新本地文档为 Draft 或 Review；只有用户审核后才能改为 Accepted。

### 事实快照规则

- `03_设计边界与规范/00_事实依据/原文快照/` 中的文件不得直接改写。
- 上游原文发生变化时，重新同步快照并更新索引中的来源、日期和 SHA-256。
- 对原文的解释、归纳和设计推导写入索引、派生规范或过程存档，不写回原文快照。
- 正式 Build 的元信息必须链接其使用的事实依据和派生规范。
- 所有表述需区分：`已确认事实`、`设计推导`、`候选`、`待验证`。

## 输出类型

每份新文档必须归入以下一种且只能有一种主类型：

1. **正式 Build 文档**：描述进入 Review 或 Accepted 的 Build 定义、组件、循环、投放与数值接口；只有 Accepted 可作为开发输入，每套 Build 只保留唯一的当前版本。
2. **过程与决策存档**：保存问题、依据、候选方案、验证结果、取舍与待办，支持后续追溯；不记录冗长的逐字思维流。
3. **设计边界与规范**：定义跨 Build 复用的硬约束、术语、事件口径、数值原则和禁止事项。

不得把三类内容长期混写在同一文件中。临时草稿完成后，应将稳定结论分别回写到正式 Build 文档或设计边界文档。

## 默认目录

```text
00_Agent输出规范/       # 输出标准和模板
01_正式Build文档/       # 每套 Build 唯一的当前 Review/Accepted 文件，不保存旧版本副本
02_过程与决策存档/     # 调研、推演、评审、实验及变更记录
03_设计边界与规范/     # 全局规则、术语、边界、数值与验收规范
  00_事实依据/          # 上游原文快照、用户确认决策和事实索引
```

## 必须执行的工作原则

- 先按“事实依据工作流”检查状态并读取本地材料；只有 Accepted 事实和规范可以进入确定性设计。
- `01_正式Build文档/` 对同一套 Build 只允许存在一个文件；文件名去除 `_V<版本>` 后视为同一 Build 标识。
- 新版本进入 `01_正式Build文档/` 时，必须在同一次修改中移除被替代版本，并修正所有指向旧文件的链接。
- Draft、方案比较和评审过程写入 `02_过程与决策存档/`；`01_正式Build文档/` 不保存仅供追溯的正式 Build 副本。
- 旧正式版本通过 Git 历史和过程文档中的变更摘要追溯，不为了保留历史而在 `01_正式Build文档/` 并列多个版本。
- 正式文档只写当前 Review 或 Accepted 结论；被替代内容进入过程存档，不在正文中堆叠历史方案。
- 所有关键数值都必须注明状态：`占位`、`待验证`、`候选` 或 `已确认`。
- 不得凭空补齐项目事实。缺少依据时必须明确写为假设、风险或待确认项。
- 任何新机制都应明确事件源、触发条件、结算时机、作用对象、叠加规则、上限、冷却和异常情况。
- 同一术语必须保持同一含义；如需改变口径，先更新边界/规范文档并记录迁移影响。
- 文档应能让策划、程序和测试分别找到其所需信息，不依赖作者口头补充才能执行。
- 修改现有文档时保留用户已有内容，除非任务明确要求重构、迁移或删除。

## 命名约定

- 正式 Build：`Build_<属性或流派>_<核心机制>_V<主版本.次版本>.md`
- 同一 `Build_<属性或流派>_<核心机制>` 在 `01_正式Build文档/` 中最多匹配一个版本文件。
- 过程存档：`YYYY-MM-DD_<主题>_<记录类型>.md`
- 设计规范：`规范_<主题>_V<主版本.次版本>.md`
- 文件名避免“最终版”“最新版”“真的最终版”等不可追踪表述。

## 交付前检查

Agent 在宣布完成前至少检查：

- 文档类型和存放目录正确；
- 标题、版本、状态、负责人/记录者、更新时间齐全；
- 正式 Build 的组件、循环、成长阶段、数值接口、边界和验收标准完整；
- 过程文档包含输入依据、方案比较、结论与未决项；
- 新增规则没有与 Accepted 边界文档冲突；
- 所有表格口径一致，公式变量可解释，待定值有明确标记；
- 文件内相对链接有效。
- 正式 Build 已链接使用的事实依据，且没有把参考候选当成已确认事实。
- 本轮新增的用户确认结论已经保存到本地，而不是只存在于对话中。
- `01_正式Build文档/` 不存在同一 Build 的多个版本，且状态只能是 Review 或 Accepted。
- 被替代正式版本已经从 `01_正式Build文档/` 移除，且所有旧版本链接已更新为当前唯一版本或 Git 历史说明。
