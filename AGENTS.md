# Roguelike Build 设计工作区：Agent 输出规则

本目录用于设计 Roguelike 局内 Build 套路、组件关系、投放结构与数值拆解。所有 Agent 在本目录新增或修改设计文档时，必须遵守 [Agent输出规范](./00_Agent输出规范/Agent输出规范.md)，并从 [事实依据索引](./03_设计边界与规范/00_事实依据/规范_事实依据索引_V1.0.md) 开始读取项目事实。

## 事实依据工作流

对话不是长期事实库。任何会影响后续设计的项目事实、用户确认结论或规则变更，必须先保存到本地事实依据或设计规范中，再用于正式 Build。

### 开始任务前

1. 先读 [事实依据索引](./03_设计边界与规范/00_事实依据/规范_事实依据索引_V1.0.md)。
2. 根据索引中的“回答范围”读取与本任务相关的本地事实原文，不要求无差别加载全部材料。
3. 再读取 `03_设计边界与规范/` 中当前生效或待评审的派生规范，确认哪些规则是已确认、候选或待验证。
4. 最后查阅相关正式 Build 和过程存档，避免重复提出已解决问题。

### 事实等级

- **F0：用户明确确认的当前决策**。保存在 `规范_事实_用户确认Build设计决策_*.md`，在对应 Build 设计范围内优先级最高。
- **F1：已批准的上游设计文档**。例如游戏愿景和核心战斗循环，是产品体验与战斗规则的权威来源。
- **F2：当前系统边界文档**。用于主动技能、被动技能、神器、进化和随机结构。
- **D：派生设计规范**。由 F0～F2 推导，必须链接依据；不得覆盖上游事实。
- **R：参考候选**。可以提供思路，不能作为项目事实引用。V1.2 神器共享网络版属于此级。

当不同来源冲突时：

1. 先按事实等级判断；
2. 同等级以更新时间更晚且范围更具体者为准；
3. 无法判断时记录冲突并向用户确认，不得静默选择；
4. 用户给出新结论后，必须在同一任务内更新本地事实摘要或规范，不能只留在聊天中。

### 事实快照规则

- `03_设计边界与规范/00_事实依据/原文快照/` 中的文件不得直接改写。
- 上游原文发生变化时，重新同步快照并更新索引中的来源、日期和 SHA-256。
- 对原文的解释、归纳和设计推导写入索引、派生规范或过程存档，不写回原文快照。
- 正式 Build 的元信息必须链接其使用的事实依据和派生规范。
- 所有表述需区分：`已确认事实`、`设计推导`、`候选`、`待验证`。

## 输出类型

每份新文档必须归入以下一种且只能有一种主类型：

1. **正式 Build 文档**：描述已进入评审或开发候选状态的 Build 定义、组件、循环、投放与数值接口；每套 Build 只保留唯一的当前版本。
2. **过程与决策存档**：保存问题、依据、候选方案、验证结果、取舍与待办，支持后续追溯；不记录冗长的逐字思维流。
3. **设计边界与规范**：定义跨 Build 复用的硬约束、术语、事件口径、数值原则和禁止事项。

不得把三类内容长期混写在同一文件中。临时草稿完成后，应将稳定结论分别回写到正式 Build 文档或设计边界文档。

## 默认目录

```text
00_Agent输出规范/       # 输出标准和模板
01_正式Build文档/       # 每套 Build 唯一的当前权威文件，不保存旧版本副本
02_过程与决策存档/     # 调研、推演、评审、实验及变更记录
03_设计边界与规范/     # 全局规则、术语、边界、数值与验收规范
  00_事实依据/          # 上游原文快照、用户确认决策和事实索引
```

## 必须执行的工作原则

- 先按“事实依据工作流”读取本地事实，再查阅 `03_设计边界与规范/` 中当前生效的规则，最后设计或修改 Build。
- `01_正式Build文档/` 对同一套 Build 只允许存在一个文件；文件名去除 `_V<版本>` 后视为同一 Build 标识。
- 新版本进入 `01_正式Build文档/` 时，必须在同一次修改中移除被替代版本，并修正所有指向旧文件的链接。
- 草稿、方案比较和评审过程写入 `02_过程与决策存档/`；`01_正式Build文档/` 不保存 `已归档`、`已废弃` 或仅供追溯的正式 Build 副本。
- 旧正式版本通过 Git 历史和过程文档中的变更摘要追溯，不为了保留历史而在 `01_正式Build文档/` 并列多个版本。
- 正式文档只写当前有效结论；被替代内容进入过程存档，不在正文中堆叠历史方案。
- 所有关键数值都必须注明状态：`占位`、`待验证`、`候选` 或 `已确认`。
- 不得凭空补齐项目事实。缺少依据时必须明确写为假设、风险或待确认项。
- 任何新机制都应明确事件源、触发条件、结算时机、作用对象、叠加规则、上限、冷却和异常情况。
- 同一术语必须保持同一含义；如需改变口径，先更新边界/规范文档并记录迁移影响。
- 文档应能让策划、程序和测试分别找到其所需信息，不依赖作者口头补充才能执行。
- 修改现有文档时保留用户已有内容，除非任务明确要求重构、迁移或删除。

## 命名约定

- 正式 Build：`Build_<属性或流派>_<核心机制>_V<主版本.次版本>.md`
- 同一 `Build_<属性或流派>_<核心机制>` 在 `01_正式Build文档/` 中最多匹配一个版本文件。
- 过程存档：`YYYY-MM-DD_<主题>_<记录类型>.md`
- 设计规范：`规范_<主题>_V<主版本.次版本>.md`
- 文件名避免“最终版”“最新版”“真的最终版”等不可追踪表述。

## 交付前检查

Agent 在宣布完成前至少检查：

- 文档类型和存放目录正确；
- 标题、版本、状态、负责人/记录者、更新时间齐全；
- 正式 Build 的组件、循环、成长阶段、数值接口、边界和验收标准完整；
- 过程文档包含输入依据、方案比较、结论与未决项；
- 新增规则没有与现行边界文档冲突；
- 所有表格口径一致，公式变量可解释，待定值有明确标记；
- 文件内相对链接有效。
- 正式 Build 已链接使用的事实依据，且没有把参考候选当成已确认事实。
- 本轮新增的用户确认结论已经保存到本地，而不是只存在于对话中。
- `01_正式Build文档/` 不存在同一 Build 的多个版本，也不存在状态为 `已归档` 或 `已废弃` 的文件。
- 被替代正式版本已经从 `01_正式Build文档/` 移除，且所有旧版本链接已更新为当前唯一版本或 Git 历史说明。
