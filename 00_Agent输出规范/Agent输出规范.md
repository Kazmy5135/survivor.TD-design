# Agent 输出规范

> 文档性质：工作区元规范  
> 版本：V1.2
> 状态：Review
> 建立日期：2026-07-20  
> 更新时间：2026-07-21  
> 适用范围：本目录内所有 Roguelike Build 设计工作

---

## 1. 目标

本规范用于把 Build 设计产物拆分为三个职责清晰、可互相引用的文档类型：

| 类型 | 回答的核心问题 | 主要读者 | 是否可直接指导开发 |
|---|---|---|---|
| 正式 Build 文档 | 这套 Build 是什么、如何运转、由什么组成？ | 策划、程序、测试 | 是 |
| 过程与决策存档 | 为什么形成当前方案、比较过什么、还缺什么证据？ | 策划、评审者、后续 Agent | 否，作为追溯依据 |
| 设计边界与规范 | 所有 Build 必须遵守什么共同规则？ | 全团队 | 是，且优先级高于单套 Build |

三类文档形成如下关系：

```text
设计边界与规范
    ↓ 约束
正式 Build 文档
    ↑ 结论沉淀
过程与决策存档
```

若正式 Build 与设计边界冲突，应先判定是 Build 越界还是全局规则需要升级，不得在单套 Build 中静默绕过规范。

### 1.1 事实依据前置

所有设计工作必须从 [事实依据索引](../03_设计边界与规范/00_事实依据/规范_事实依据索引_V1.0.md) 开始。

- 先检查文档状态；只有 Accepted 事实和规范可以进入确定性设计输入；
- 原文快照用于保存上游事实，不直接改写；
- 用户明确确认的长期结论必须先写入 F0 决策文档的 Draft/Review，并在用户审核后才成为 Accepted；
- 派生规范必须能链接到 F0～F2 事实来源；
- 正式 Build 必须区分已确认事实、设计推导、候选数值和待验证项；
- 参考候选只能用于方案比较，不能伪装成当前规则；
- 不允许把只存在于对话中的信息当作后续任务默认已知条件。

## 2. 类型一：正式 Build 文档

### 2.1 定位

Accepted 的正式 Build 文档是某套玩法套路的唯一当前有效定义。Review 版本只用于审核，不得作为开发输入。正式 Build 最终必须能被转化为：

- 功能需求；
- 配置表字段；
- 数值表；
- 测试用例；
- 投放与平衡验证计划。

正式文档不承担长篇记录探索历史的职责。

### 2.1.1 唯一当前版本规则

`01_正式Build文档/` 是 Review/Accepted 正式交付目录，不是历史版本仓库。

- 同一套 Build 只能存在一个当前 Review 或 Accepted 文件；文件名去掉 `_V<主版本.次版本>` 后作为 Build 唯一标识。
- 新版本替代旧版本时，新增当前文件与移除旧文件必须在同一次修改中完成。
- 旧版本不移动到 `01_正式Build文档/` 的子目录，也不继续并列保存。
- 草稿、候选比较、评审意见和被否决内容进入 `02_过程与决策存档/`。
- 历史正式内容由 Git 历史保留；过程文档只记录变更原因、影响和结论，不复制整份旧正式文档。
- 删除旧文件后，必须更新事实索引、过程存档、规范和其他 Build 中的旧链接。

### 2.2 必填内容

1. **文档元信息**：版本、状态、作者/维护者、更新时间、关联规范、关联过程记录。
   同时必须列出本 Build 使用的事实依据。
2. **一句话定义**：玩家做什么，形成什么循环，得到什么战斗结果。
3. **设计目标**：体验目标、公共输入、价值出口和与同属性其他 Build 的差异。
4. **Build 身份**：属性、流派、关键词、强弱场景、操作强度、成型速度和长线接口。
5. **战场信息与公共状态**：明确玩家能看到什么、不能看到什么，以及状态的统一语义。
6. **事件与结算**：触发事件、结算顺序、作用对象、频率、冷却、叠加、上限和死亡处理。
7. **主动技能与进化**：基础形态、第一次进化、路线中间 Perk、第二次进化和技能职责。
8. **被动技能**：等级效果、作用范围、进化接口、槽位占用和跨 Build 复用。
9. **神器候选**：单件价值、支持技能、支持流派、作用范围、上限和防递归。
10. **核心循环**：输入、状态变化、价值出口与回流；必须明确最小闭环。
11. **Build 完成度**：至少包含两个乞丐版入口、胡牌版和大胡/完全体。
12. **非唯一解校验**：逐项说明缺少核心技能、被动或神器时的替代路径。
13. **数值拆解接口**：阶段体验目标、输出预算、公式、变量和自上而下/自下而上双向校验。
14. **投放与赛季扩展**：共享池关系、当前未覆盖进化、未来神器切入口和旧构筑保留价值。
15. **验收与未决项**：功能、信息、数值、长线验收、失败判据和阻塞问题。

### 2.3 组件描述标准

每个组件至少使用以下字段：

| 字段 | 要求 |
|---|---|
| 名称/代号 | 唯一、稳定、可被表格引用 |
| 角色 | 启动、核心、放大、收口、容错或通用 |
| 单件价值 | 不依赖未来组件时能提供什么 |
| 触发条件 | 可被程序监听的事件与条件 |
| 效果 | 对象、数值、持续时间和作用范围 |
| 结算规则 | 时机、顺序、叠加、上限、冷却 |
| 协同关系 | 支持哪些组件或 Build，为什么 |
| 冲突与边界 | 不生效、降效或需要特殊处理的情况 |
| 数值状态 | 占位、待验证、候选或已确认 |

### 2.4 数值书写标准

- 公式先于具体数值，例如：`单次收益 = 基础值 × 层数 × 触发修正`。
- 每个变量必须说明单位、取值范围和数据来源。
- 区分期望值、上限值、实测值和配置值。
- 百分比需明确是加法、乘法还是独立乘区。
- 概率需明确每次触发独立判定、共享冷却或伪随机规则。
- 暂未拍定的数字统一写为 `[候选：10%～15%]` 或 `[待验证]`，不得伪装成最终值。
- 先定义乞丐、成型、胡牌和大胡的阶段目标，再把强度预算拆到技能、被动和神器。
- AOE、持续伤害、引爆和资源收益必须放入同一价值预算，禁止分别按满价值相加。
- 必须用实际命中目标数、覆盖率、触发率和能耗从下往上反算，并与阶段目标比较误差。

### 2.5 不应出现

- 未注明来源的“感觉会很好玩”；
- 只有背景故事，没有可执行机制；
- 组件名称堆砌，但没有输入—输出闭环；
- 把唯一固定配方伪装成开放构筑；
- 依赖口头解释才能知道的触发和结算规则；
- 大段保留已废弃方案。

模板见：[正式RoguelikeBuild文档模板](./模板_正式RoguelikeBuild文档.md)。结构以当前 [火系灼烧爆破 V0.2](../01_正式Build文档/Build_火_灼烧爆破_V0.2.md) 为基准，后续具体 Build 不得降级为只有神器组合和战斗想象的简表。

## 3. 类型二：过程与决策存档

### 3.1 定位

过程文档用于保存可复核的设计轨迹，而非正式规则本身。它记录：

- 收到的需求和输入材料；
- 已知事实、设计假设和未知项；
- 候选方案及比较维度；
- 数据、测试或案例证据；
- 关键取舍与决策理由；
- 被否决方案及否决原因；
- 风险、待办和后续验证计划。

“思考过程”在本工作区中定义为**结构化设计依据和决策记录**，不要求保存逐字的内部推演或无结论的意识流。

### 3.2 记录类型

文件名结尾应标注以下一种记录类型：

- `需求拆解`
- `方案探索`
- `数值推演`
- `评审记录`
- `测试记录`
- `变更记录`
- `问题调查`

### 3.3 必填内容

1. 记录目标与范围；
2. 输入材料及其版本；
3. 已知事实 / 假设 / 未知项分栏；
4. 候选方案；
5. 统一的比较标准；
6. 证据或验证结果；
7. 当前结论及其置信度；
8. 对正式文档或规范文档的影响；
9. 未决问题、下一步、负责人；
10. 关联文档链接。

### 3.4 决策记录格式

关键决定统一使用：

```text
决策：选择什么
原因：基于哪些目标、事实或测试
代价：牺牲了什么
替代方案：还比较过什么
适用条件：在什么前提下成立
复审触发器：出现什么证据时重新评估
```

### 3.5 生命周期

- 过程文档同样只使用 Draft、Review、Accepted 三种状态。
- “被后续方案替代”“仅用于历史追溯”“问题已失效”等信息写入单独的 `结果说明` 字段，不再充当状态。
- 过程文档即使 Accepted，也只表示其决策记录已被审核；能否作为设计依据仍取决于门禁中是否明确把它列为前置文档。

模板见：[过程与决策存档模板](./模板_过程与决策存档.md)。

## 4. 类型三：设计边界与规范

### 4.1 定位

边界与规范文档定义跨 Build 生效的稳定规则。其优先级高于单套正式 Build 文档，主要包括：

- 核心术语与事件定义；
- Build 结构原则；
- 组件设计硬约束；
- 数值预算和乘区规则；
- 投放、稀有度和共享池规则；
- 叠加、触发、冷却与结算顺序；
- 体验底线、禁区和例外审批；
- 开发与测试验收口径。

### 4.2 规则等级

每条规则必须标注等级：

| 等级 | 含义 | 处理方式 |
|---|---|---|
| MUST | 硬约束 | 不满足则不得进入开发；例外必须单独审批 |
| SHOULD | 默认原则 | 允许有理由地偏离，并记录影响 |
| MAY | 可选建议 | 按 Build 目标决定是否采用 |
| FORBIDDEN | 禁止事项 | 不得使用，除非规范本身升级 |

### 4.3 单条规则格式

每条规则至少包含：

```text
规则 ID：稳定且唯一，例如 BUILD-COMP-001
等级：MUST / SHOULD / MAY / FORBIDDEN
规则正文：可判断是否满足的一句话
目的：保护什么体验或技术目标
适用范围：影响哪些 Build、组件或事件
验证方式：如何在文档、配置或实机中检查
例外流程：谁批准、记录在哪里、何时复审
```

### 4.4 版本与变更

- 修改术语含义、事件口径或硬约束时，至少提升次版本。
- 删除或反转 Accepted 规则时，必须记录受影响的正式 Build。
- 新规则从文档标注的生效版本开始约束后续设计。
- 如需兼容旧内容，明确“旧版本继续有效”还是“必须迁移”。

### 4.5 禁止含糊表述

规范中避免使用无法验收的词：

- “适量”“较高”“偶尔”“尽可能”；
- “一般情况下”“正常来说”；
- “体验要爽”“不要太强”。

若暂时无法量化，应给出判定案例、对照基线或待验证指标。

模板见：[设计边界与规范模板](./模板_设计边界与规范.md)。

## 5. 三类文档之间的信息流

### 5.1 新设计

1. 检查当前节点要求的前置文档是否全部 Accepted。
2. 若不满足，当前节点保持 Blocked，不开始设计。
3. 满足后，在 Draft 中拆解需求、记录假设并比较候选方案。
4. 讨论形成完整初版后，将输出状态改为 Review，提交设计负责人审核。
5. 只有设计负责人明确接受后，文档才改为 Accepted，并解锁下一节点。
6. 若出现 D1 未定义语义，立即暂停并回到 D1 走 Draft→Review→Accepted。

### 5.2 设计变更

1. 过程存档以 Draft 记录变更原因、影响范围和候选处理方式。
2. 旧 Accepted 在新版通过前仍是当前依据；新版不得提前覆盖其语义。
3. 新版进入 Review 后列出全部受影响下游文档。
4. 新版 Accepted 后，受语义影响的下游文档自动降为 Review并重新审核。
5. 正式文档版本递增；旧版本由 Git 历史保留，只保留一个当前文件。

### 5.3 数值分解

1. 正式 Build 定义机制公式、变量和目标表现。
2. 过程存档保存样本、推演、模拟与测试结果。
3. 达到确认条件后，将最终区间或数值回写正式 Build。
4. 跨 Build 共用的预算、乘区和上限进入数值规范。

## 6. 文档状态、引用与版本

### 6.1 状态

状态只允许以下三种：

- `Draft`：讨论中；不能作为下游依据或开发输入。
- `Review`：已形成初版，等待设计负责人审核；不能解锁下一节点。
- `Accepted`：设计负责人已审核接受；可以被下游引用并解锁门禁。

状态推进顺序固定为：`Draft → Review → Accepted`。Codex 无权自行把 Review 改为 Accepted。

详细引用与回退规则见 [文档状态与设计门禁](../03_设计边界与规范/规范_文档状态与设计门禁_V0.1.md)。在该规范本身 Accepted 前，现有文档统一视为 Review。

### 6.1.1 引用权限

- “事实依据、前置条件、关联规范、生效规则”只能引用 Accepted 文档。
- Draft/Review 只能用于讨论、比较和历史追溯，并必须标注为非生效依据。
- 下游发现 D1 缺失语义时必须停止，回到 D1 补文档并重新通过门禁。

### 6.2 版本

- 主版本变化：核心循环、组件结构、事件语义或关键边界发生不兼容改变。
- 次版本变化：新增组件、补充规则、数值区间调整或不改变核心循环的完善。
- 文字修订：错字、排版、链接修复可不升版本，但应更新日期。

## 7. 统一质量门槛

文档进入 Review 前，应满足：

- **可理解**：新读者能复述 Build 循环和主要决策。
- **可实现**：程序能识别所需事件、字段、顺序和异常处理。
- **可配置**：数值变量有名称、单位、范围和状态。
- **可测试**：测试能从验收条目构造明确用例。
- **可追溯**：关键结论能链接到过程依据或规范规则。
- **有事实来源**：项目事实来自本地事实依据，新的用户确认已完成本地沉淀。
- **可维护**：当前结论与历史探索分离，版本与状态明确。
- **唯一交付**：每套 Build 在 `01_正式Build文档/` 中只有一个当前版本，旧版本仅从 Git 历史追溯。
- **可比较**：不同 Build 使用一致的强项、弱项和数值维度。
