## 1. 状态与文档类型分离

“正式 Build 文档、过程与决策存档、设计边界与规范”是文档类型；`Draft / Review / Accepted` 是文档状态。任意一种文档类型都必须处于三种状态之一。

## 2. 三种状态

| 状态 | 定义 | 可以做什么 | 不可以做什么 |
|---|---|---|---|
| Draft | 正在讨论、结构或结论尚未稳定 | 记录问题、候选、假设和讨论结论 | 不能作为任何下游设计的事实依据、前置条件或开发输入 |
| Review | 讨论已经形成初版，等待设计负责人审核 | 供审核、批注、比较和修订 | 不能作为后续阶段的设计依据，不能解锁门禁，不能指导开发 |
| Accepted | 设计负责人已明确审核接受 | 可以被下游文档引用，可以解锁依赖它的设计阶段 | 不能被下游文档静默改写；变更必须重新走 Draft→Review→Accepted |

状态只能按以下方向推进：

```text
Draft → Review → Accepted
```

- Codex 可以将 Draft 整理为 Review。
- 只有设计负责人可以通过明确审核结论把 Review 改为 Accepted。
- Codex 不得根据“看起来没有问题”自行标记 Accepted。

## 3. 引用规则

### DOC-REF-001

- 等级：MUST
- 规则正文：下游文档的“事实依据、前置条件、关联规范、生效规则”只能引用 Accepted 文档。

### DOC-REF-002

- 等级：MUST
- 规则正文：Draft 或 Review 文档可以在讨论记录、问题比较或历史追溯中被链接，但必须明确标注“非生效依据”，且不能据此继续下一阶段设计。

### DOC-REF-003

- 等级：FORBIDDEN
- 规则正文：不得因为文件位于 `01_正式Build文档/`、文件名带“正式”或正文写了“已确认事实”，就绕过状态门禁；状态栏是唯一准入判断。

### DOC-REF-004

- 等级：MUST
- 规则正文：一个文档如同时引用 Accepted 与非 Accepted 输入，非 Accepted 输入只能作为候选或风险，不能进入该文档的确定结论。

## 4. 设计门禁

每个设计节点必须声明：

- 节点编号；
- 前置 Accepted 文档；
- 本节点输出文档；
- Review 检查项；
- Accepted 判定者；
- 解锁的下一节点。

只有全部前置文档处于 Accepted，节点才允许从“Blocked”进入“Draft”。节点输出只有全部变为 Accepted，才允许解锁下一节点。

```text
前置文档全部 Accepted
→ 当前节点允许 Draft
→ 讨论形成 Review
→ 设计负责人审核
→ Accepted
→ 解锁下一节点
```

## 5. D1 缺失语义回补规则

### GATE-D1-001

- 等级：MUST
- 规则正文：后续讨论中一旦发现当前设计需要使用 D1 未定义的术语、事件、状态或结算规则，必须暂停当前节点。

Codex 必须按以下顺序处理：

1. 明确告诉设计负责人“当前缺少哪条 D1 语义”；
2. 说明缺失会影响哪些正在设计的内容；
3. 不在下游文档中临时补定义；
4. 回到 D1 文档新增或修订该规则，状态从 Draft 开始；
5. D1 修订达到 Review 后提交设计负责人审核；
6. 只有新版 D1 Accepted 后，才恢复被阻塞的下游设计。

### GATE-D1-002

- 等级：SHOULD
- 规则正文：D1 初版只定义当前已知会被宏观设计使用的最小语义集合，不为了追求一次性完备而预先发明大量规则。

## 6. Accepted 文档变更

1. 先建立变更记录和新版 Draft；旧 Accepted 版本在新版通过前仍是当前依据。
2. 新版进入 Review 后，列出受影响的所有下游 Accepted 文档。
3. 设计负责人接受新版后，受语义影响的下游文档自动降为 Review，必须重新审核。
4. 只修改错字、排版或无语义链接时可以不让下游降级，但仍需记录。
5. 正式 Build 的唯一当前版本规则继续适用；旧版本由 Git 历史保留。

## 7. 过渡规则

- 本规范建立时，工作区现有受控设计文档全部标记为 Review。
- 这些 Review 文档保留已有内容用于审核和重整，但不再自动构成后续设计依据。
- 本条同时作为过渡批次的统一引用标识：现有 Review 文档中的“事实依据、关联规范、关联 Build”等链接全部视为非生效审核链接，无需逐份改写；新建文档不适用此豁免。
- 在本规范和新规划路线被设计负责人标记为 Accepted 前，D1 不开始。
- 原文快照不直接修改，不进入 Draft/Review/Accepted 状态机；它们是只读来源。使用其中结论时，必须先形成一份受控设计文档并达到 Accepted。
- `AGENTS.md`、模板结构和工具说明属于工作区治理/执行文件，不作为游戏设计结论；模板本身仍在本轮统一标记为 Review 以便审核。

## 8. Review 检查清单

- [ ] 文档状态只使用 Draft、Review、Accepted。
- [ ] 所有确定性引用指向 Accepted 文档。
- [ ] Draft/Review 链接均标注为非生效依据。
- [ ] 文档声明了前置门禁和解锁对象。
- [ ] D1 缺失语义会触发回补，不在下游临时定义。
- [ ] Accepted 变更已列出下游降级影响。
- [ ] 原文快照没有被改写。
