## 1. 规范目的

建立低理解成本、可稳定启动、同时保留胡牌上限的 Build 设计边界，并让可叠层元素 Debuff 在 400～500 个敌人的战场中仍然能够被表现、理解和验证。

本规范依据 F0～F2 事实派生。关键输入包括：

- 单局 8～12 分钟，以主动释放技能和割草反馈为核心；
- 主动技能无冷却，只检查技能费用；
- 主动技能、被动技能和神器职责分离；
- 神器每局获得 2～3 件且不升级；
- 场上约有 400～500 个敌人，不展示逐个敌人的 Debuff 数值与来源；
- 禁止秒杀、永久控制、屏幕特效过载和高理解成本的递归转换；
- V1.2 的热量、电荷、寒能与固定神器核心双件仅为参考反例。

## 2. Build 的基本单位

### BUILD-STRUCT-001

- 等级：MUST
- 规则正文：一套 Build 必须由 1～3 个主动技能、相关进化路线、支持被动和 1～3 件神器共同描述，不能只把某个神器组合称为完整 Build。
- 验证方式：正式 Build 文档必须列出乞丐版、胡牌版和大胡/完全体。

### BUILD-STRUCT-002

- 等级：MUST
- 规则正文：每套 Build 必须至少存在以下三个完成度。

| 完成度 | 主动技能参与数 | 被动/进化 | 神器状态 | 体验要求 |
|---|---:|---|---|---|
| 乞丐版 | 1～2 | 核心技能能够进入对应路线 | 至少 1 件有效 | 能正常运转并维持基础战力 |
| 胡牌版 | 2～3 | 两个技能或路线形成明确分工 | 2 件产生自然联动 | 出现明显效率或玩法变化 |
| 大胡/完全体 | 2～3 | 相关进化与被动均指向同一循环 | 2～3 件共同支持 | 接近理论上限但仍需要操作 |

### BUILD-STRUCT-003

- 等级：FORBIDDEN
- 规则正文：不得让 Build 只能通过唯一的技能 B、被动 B 或神器 B 才能启动。
- 验证方式：每套 Build 至少给出两个乞丐版入口，并说明缺少某个组件时的替代路径。

### BUILD-STRUCT-004

- 等级：MUST
- 规则正文：完全体 Build 必须有 2～3 个主动技能产生机制关联；只强化一个主动技能的内容可以成为乞丐版入口，但不能构成正式 Build 的全部内容。

## 3. 神器规则

### ARTIFACT-001

- 等级：MUST
- 规则正文：所有神器在系统地位和设计规格上等价，不划分“立即有效神器”和“联动神器”等隐性等级。

### ARTIFACT-002

- 等级：MUST
- 规则正文：每件神器单独获得时必须产生完整且可说明的规则变化，不依赖另一件指定神器才能生效。

### ARTIFACT-003

- 等级：MUST
- 规则正文：每次神器三选一至少有两个选项能够作用于玩家当前持有的主动技能或被动技能。
- 补充：第三个选项可以是通用强化或合理后续方向，但不得是本局以后也难以触发的无效项。

### ARTIFACT-004

- 等级：SHOULD
- 规则正文：神器优先读取公开且直观的技能标签或战斗行为，例如施加燃烧、命中燃烧目标、Debuff 满层、清除全部 Debuff、击杀或消耗能量；不为单个技能维护隐藏适配表。

### ARTIFACT-005

- 等级：FORBIDDEN
- 规则正文：不得为了让神器之间产生联动而创造只用于计数和换算的内部资源。
- 当前明确废弃：热量、电荷、寒能。

## 4. 主动技能释放口径

### SKILL-CAST-001

- 等级：MUST
- 规则正文：技能设计以“一次完整释放”为基本事件；常驻释放只是费用足够时自动重复执行下一次相同释放的便捷操作。

```text
开始一次释放
→ 检查并扣除本次费用
→ 执行技能
→ 本次释放结束
→ 若常驻开启且费用足够，则自动开始下一次释放
```

### SKILL-CAST-002

- 等级：FORBIDDEN
- 规则正文：常驻不能绕过费用检查，也不因为自动重复而获得额外伤害、状态或神器收益。

## 5. 战场信息表达

### READABILITY-001

- 等级：MUST
- 规则正文：所有 Build 必须按同屏约 400～500 个敌人、无法展示逐个 Debuff 数字的条件设计。
- 验证方式：玩家不查看单个敌人的精确层数，也能理解技能的主要用途和结果。

### READABILITY-002

- 等级：FORBIDDEN
- 规则正文：不得要求玩家识别某个敌人当前是 3 层、5 层或 7 层 Debuff 后再做操作决策。

### READABILITY-003

- 等级：MUST
- 规则正文：同一种 Debuff 在单个敌人身上只使用一个统一状态表现，不按施加技能、层数来源或伤害来源拆分图标和特效。

### READABILITY-004

- 等级：SHOULD
- 规则正文：层数相关玩法优先使用玩家能够通过状态变化理解的语义：未施加、已施加、满层、全部清除。
- 补充：满层如需强化表现，应复用同一个 Debuff 特效的强度或阶段变化，不新增逐层图标。

## 6. 元素 Debuff 公共规则

### DEBUFF-COMMON-001

- 等级：MUST
- 规则正文：火、电、冰三种基础 Debuff 均使用可叠层结构，为技能、Perk、神器和局外成长保留数值空间。

### DEBUFF-COMMON-002

- 等级：MUST
- 规则正文：每次成功施加 Debuff 时，增加效果来源规定的固定层数，并把整组 Debuff 的剩余持续时间刷新到标准持续时间。

```text
结算后层数 = min（当前层数 + 本次施加层数，当前层数上限）
结算后持续时间 = 当前标准持续时间
```

超过上限的部分默认丢弃，不转换为另一种资源。

### DEBUFF-COMMON-003

- 等级：SHOULD
- 规则正文：承担基础挂状态职责的技能优先使用确定性层数，不用低概率挂 1 层作为主要成长接口。

### DEBUFF-COMMON-004

- 等级：MUST
- 规则正文：所有伤害必须区分直接伤害、持续伤害和规则触发伤害；持续伤害不能暴击。

### DEBUFF-COMMON-005

- 等级：MUST
- 规则正文：层数上限、单次施加层数、持续时间和每层强度均为独立的全局配置字段。
- 局外成长可以使用的维度：提高单次施加层数、持续时间、每层效果，以及在 UI 与数值允许时提高层数上限。

### DEBUFF-SOURCE-001

- 等级：MUST
- 规则正文：同一目标身上同一种 Debuff 的所有层均为同质层，使用相同的每层效果、持续时间和结算规则，不保留玩家不可见的技能来源差异。
- 目的：避免 A 技能的燃烧每层 10 伤害、B 技能的燃烧每层 20 伤害，却只显示一个燃烧特效。

### DEBUFF-SOURCE-002

- 等级：FORBIDDEN
- 规则正文：单个主动技能的路线 Perk 不得只修改由该技能施加的某种共享 Debuff 层的伤害、持续时间或其他层属性。
- 允许的技能局部成长：直接伤害、范围、费用、命中频率、每次施加层数，以及对“已施加/满层”目标产生的额外技能效果。

### DEBUFF-SOURCE-003

- 等级：MUST
- 规则正文：若效果修改某种 Debuff 的每层伤害或其他共享层属性，必须对该 Debuff 的全部来源统一生效，并放在神器、全局被动或系统级成长中。

### DEBUFF-THRESHOLD-001

- 等级：FORBIDDEN
- 规则正文：当前版本不得使用“达到 X 层”且 X 小于当前层数上限的中间阈值作为玩家效果条件。

### DEBUFF-THRESHOLD-002

- 等级：MUST
- 规则正文：需要层数阈值时使用“达到当前层数上限/满层”，不得把效果写死为 5 层、7 层等数字。
- 目的：玩家无需读取具体层数；局外提高层数上限后，规则语义仍然成立。

### DEBUFF-THRESHOLD-003

- 等级：MAY
- 规则正文：层数可以在后台作为连续倍率或转移数量使用，例如“每清除 1 层提高伤害”或“继承目标当前层数”；前提是中间层数不产生离散分支，也不要求玩家知道精确数字才能操作。
- 目的：保留叠层的数值成长价值，同时避免把不可见数字变成操作门槛。

### DEBUFF-CONSUME-001

- 等级：MUST
- 规则正文：消耗 Debuff 的效果必须清除目标当前全部层数；收益可以按实际清除层数计算。

```text
本次清除层数 = 结算前当前全部层数
结算后层数 = 0
收益 = 本次清除层数 × 每层收益
```

### DEBUFF-CONSUME-002

- 等级：FORBIDDEN
- 规则正文：不得从 7 层中只消耗 5 层并留下 2 层，也不得让玩家在看不到层数时选择消耗数量。

### DEBUFF-CONSUME-003

- 等级：MUST
- 规则正文：同一个消耗事件清除 Debuff 后，不得在该事件尾部立即重新施加同一种 Debuff；状态特效必须实际消失。
- 目的：让“清除/引爆”在视觉和规则上都产生可确认的状态切换。

## 7. 三种基础 Debuff

以下层数上限均以 `10 层` 作为首轮测试候选，不是已确认最终值。

### 7.1 火：燃烧

```text
每层燃烧按固定间隔造成一次火属性持续伤害。
燃烧总伤害随当前层数线性增长。
```

- 再次施加：增加固定层数并刷新整组持续时间；
- 伤害类型：持续伤害，不可暴击；
- 所有技能施加的燃烧层使用相同的每层伤害；
- 基础身份：时间驱动的稳定持续伤害；
- 可扩展事件：施加燃烧、燃烧存在、燃烧满层、清除全部燃烧、燃烧击杀、燃烧结束。

### 7.2 电：感电（候选，暂不展开 Build）

```text
感电目标受到直接伤害时，额外触发一次电击伤害。
电击伤害随当前感电层数提高。
```

- 再次施加：增加固定层数并刷新整组持续时间；
- 触发：目标受到直接伤害；
- 限制：同一目标存在短触发间隔；感电触发的电击不能再次触发感电；
- 基础身份：由后续命中频率驱动的响应伤害；
- 可扩展事件：施加感电、触发电击、感电满层、清除全部感电、连锁到其他目标。

### 7.3 冰：寒冷（候选，暂不展开 Build）

```text
每层寒冷降低目标一定比例的移动速度。
寒冷满层时触发一次短暂冰冻，并清除全部寒冷。
```

- 再次施加：增加固定层数并刷新整组持续时间；
- 基础身份：通过叠层延缓敌人推进，并在满层时获得短控制窗口；
- 冰冻限制：冰冻期间不能刷新冰冻；结束后获得短暂冰冻免疫；Boss 的冰冻时间缩短；
- 可扩展事件：施加寒冷、寒冷满层、清除全部寒冷、触发冰冻、冰冻结束。

冰冻是寒冷满层后的控制结果，不作为第四种可叠层 Debuff。

## 9. 复杂度和表现边界

### COMPLEXITY-001

- 等级：FORBIDDEN
- 禁止：秒杀、永久控制、无限资源循环、同一价值重复完整转换、需要连续解释三层以上才能说明的收益链。

### COMPLEXITY-002

- 等级：MUST
- 规则正文：一条效果应尽量通过“一个条件＋一个主要结果”表达；必要的上限、冷却和防递归属于结算说明，不应成为玩家理解效果的主体。

### VFX-001

- 等级：MUST
- 规则正文：范围强化优先复用已有弹道、火柱、命中特效和爆炸表现；不得为每层 Debuff、每个来源或每次触发生成独立大面积特效。

## 10. 正式 Build 检查清单

- [ ] 不要求玩家读取某个敌人的精确 Debuff 层数。
- [ ] 没有使用小于上限的中间层数阈值。
- [ ] 若内部使用当前层数计算倍率，玩家不需要知道精确层数也能理解“层数越多，收益越高”。
- [ ] 消耗 Debuff 时清除全部层数。
- [ ] 清除后同一事件不会立即重新施加同种 Debuff。
- [ ] 同种 Debuff 不按主动技能来源保存不同的每层效果。
- [ ] 技能局部 Perk 没有修改只有该技能来源的共享 Debuff 层属性。
- [ ] 全局 Debuff 强化明确放在神器、全局被动或系统成长中。
- [ ] 效果在 400～500 个敌人的同屏规模下仍能用统一特效表达。
- [ ] 乞丐版、胡牌版和大胡/完全体均有可执行路径。
- [ ] 不存在必须等待唯一技能、被动或神器的启动结构。
