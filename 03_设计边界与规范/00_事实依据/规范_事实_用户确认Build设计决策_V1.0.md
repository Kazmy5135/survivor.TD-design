## 1. V1.2 的定位

- 《局内 Build 设计方案与候选 Build 库 V1.2》是参考候选，不是现行方案。
- 不满意 V1.2 的主要原因是概念过多、理解与表达成本过高。
- 可以保留“不能选 A 后只剩唯一 B”这一问题意识，但不继承其内部储存资源和固定核心双件结构。

## 2. 目标构筑体验

- 优先保证容易看懂、稳定成型。
- 不要求高频转向，但不能因为选不到唯一指定组件导致 Build 崩溃。
- 不刻意追求两个看似无关神器产生隐藏联动的探索感。
- 可以同时存在高操作和低操作 Build。
- 不追求极端数值爽感，只要求玩家能明显感受到变强。
- 常规局单件神器能对对应技能产生收益；约 3～4 局出现一次多件神器和技能完美吻合的胡牌体验。
- 禁止秒杀、永久控制、屏幕特效过载、高理解成本和“左脚踩右脚”式多层转换。

## 3. Build 完整度

- 正式设计的 Build 不能只与一个主动技能产生联动。
- 乞丐版允许只有 1 个核心技能，或 1～2 个相关技能。
- 完全体需要 2～3 个主动技能与同一 Build 产生关联。
- Build 存在胡牌与大胡区别：技能、进化、被动和神器吻合程度越高，完成度越高。
- 缺少某个唯一技能、被动或神器时仍应存在可玩的替代路径。

## 4. 神器

- 所有神器在系统地位上等价，不划分“立即有效神器”和其他神器。
- 每件神器单独获得时都应有完整价值。
- 多神器联动是胡牌上限，不是神器基础生效的前提。
- 每次神器三选一至少有两个选项能够作用于当前主动技能或被动技能。
- 神器不应通过热量、电荷、寒能一类中间计数资源建立联动。

## 5. 技能释放与常驻

- 设计逻辑仍以一次完整技能释放为单位。
- 常驻只是便捷操作：一次技能结束时，如果费用足够就自动再释放一次。
- 所有技能都可以开启该自动重复功能。
- Build 设计默认不需要围绕“常驻”建立新概念；自动与手动释放使用同一套规则。

## 6. 元素 Debuff

- 火、电、冰三种基础 Debuff 都可以叠层，为局内和局外成长保留空间。
- 再次施加会增加层数，并刷新持续时间。
- 层数上限、单次施加层数、每层效果和持续时间可以分别成长。
- 持续伤害不能暴击。
- AOE 技能容易命中多个目标，因此主要挂状态接口不应依赖“20% 概率挂 1 层”一类弱反馈方式。
- 火、电、冰的跨属性联动暂不设计，等单属性流程跑通后再处理。

## 7. 抽象资源

以下 V1.2 概念明确废弃：

- 热量；
- 电荷；
- 寒能。

除非一个新资源容易理解、容易显示、会让玩家主动决策且无法用现有技能、能量、Debuff 或战斗事件表达，否则不引入新的计数资源。

## 8. 当前已有火属性主动技能

### 火焰喷射器

- 发射一条火柱；
- 对火柱经过的敌人每 0.5 秒造成一次火属性伤害；
- 当前只完成基础表现；
- 后续进化可以在 Build 设计中自由提出。

### 导弹

- 发射一枚导弹；
- 对命中目标造成一次性伤害；
- 对命中目标周围敌人造成一次 AOE 伤害；
- 当前只完成基础表现；
- 后续进化可以在 Build 设计中自由提出。

## 9. 被动技能容量

- 当前总共只有 8 个被动技能。
- 其中 3 个被动技能需要分配给能量相关方向。
- 火属性专属或强相关被动只能占用 1～2 个名额。
- 设计火 Build 时应优先复用能量、范围、Boss/单体等通用被动接口，避免单套 Build 独占过多被动槽位。

## 10. 尚未确认、不得当作事实的内容

- 三种 Debuff 的最终层数上限；`10 层` 目前只是首轮测试候选。
- 火焰喷射器和导弹每次施加的具体燃烧层数。
- 燃烧持续时间、跳伤间隔和每层伤害。
- 感电与寒冷的最终基础效果。
- 导弹是否消耗燃烧层数以及具体消耗方式。
- 任意主动技能进化、被动技能和神器的最终名称、效果与数值。
- 跨属性 Build 与属性反应。

这些内容必须在派生规范或正式 Build 中标为候选/待验证，经过确认后再升级为 F0 事实。
