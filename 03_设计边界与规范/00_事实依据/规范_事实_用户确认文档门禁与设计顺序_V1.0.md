# 用户确认的文档门禁与设计顺序 V1.0

> 文档类型：设计边界与规范  
> 事实等级：F0  
> 状态：Review  
> 确认者：设计负责人  
> 记录者：Codex  
> 创建日期：2026-07-21  
> 更新时间：2026-07-21  
> 适用范围：文档状态、引用权限、D1 回补、MVP 玩法内容规划和数值设计顺序

---

## 1. 文档状态

- 设计文档只使用 Draft、Review、Accepted 三种状态。
- Draft 表示讨论过程中的草稿。
- Review 表示已经完成讨论并形成初版，等待设计负责人审核。
- Accepted 表示设计负责人已经审核接受。
- 只有 Accepted 文档可以被后续设计引用并解锁下一节点。
- 当前已有受控设计文档统一标记为 Review。

## 2. D1 基础语义

- D1 负责定义战斗基础语义和规则底座。
- D1 初版不需要一次考虑全面，只定义当前宏观设计需要使用的最小语义集合。
- 后续讨论如果出现 D1 未定义的语义或规则，Codex 必须指出缺失并要求先补充 D1 文档。
- D1 补充内容需要重新经过 Draft→Review→Accepted；未 Accepted 前，受影响的下游设计暂停。

## 3. 玩法内容设计顺序

采用从宏观向下拆解：

1. 先设计 MVP 版本全部流派、方向和定位；
2. 再拆解每条流派需要的武器类型定位，包括每个进化的定位；
3. 同阶段设计具体神器及其 Build 职责；
4. 同时考虑后续赛季新增武器、神器和上位 Build 的扩展空间；
5. 只有上述内容 Accepted 后，才设计具体武器表达、基础形态、进化和被动绑定。

武器职责与具体表达必须分离。同一“火属性高频 AOE”职责可以由火焰力场表达，也可以由范围进化后的火焰喷射器表达。

## 4. 抽象单局数值线

完成具体内容机制后，开启独立的单局体验规划线。该规划先把玩家实力抽象为有效 DPS、清场能力和生存容错，不考虑具体使用哪条 Build。

需要先规划：

- 关卡总波次；
- 每波持续时间；
- Boss 数量、位置和战斗时长；
- 每波 NPS 与对应压力；
- 玩家总成长等级和成长节点；
- 宝箱数量、掉落时间和成长价值；
- 不同阶段的抽象有效 DPS 需求。

此阶段不包含具体技能、升级或神器数值。

## 5. 具体数值拆解顺序

- 抽象单局体验曲线 Accepted 后，先让每条套路在不同阶段的有效 DPS 与曲线对齐。
- Build 阶段价值 Accepted 后，才拆解每个武器、每次升级、进化、被动和神器的具体数值。
- 不允许先给单件内容填数，再反向拼凑整局体验。
